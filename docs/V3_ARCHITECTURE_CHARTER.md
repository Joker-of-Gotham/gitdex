# GitDex V3 架构宪章

## 1. 目标

GitDex V3 采用 Big Bang 一次性切换，目标是建立可演进、可诊断、可回放、跨平台一致的 AI 原生 Git 工作台。

V3 的主目标：

- 把流程控制从“分散 if-else”升级为显式状态机。
- 把执行层从“命令直调”升级为“runtime + adapter + 诊断 + 恢复”。
- 把跨层数据从“直接结构体耦合”升级为版本化契约。
- 把 UI 从“单页拼接”升级为可组合 Section 组件与统一交互引擎。
- 把可观测性从“零散日志”升级为 trace/log/metrics 一体化链路。

## 2. 架构边界

V3 明确三平面边界：

1) 控制平面（Control Plane）
- 负责：配置生效、角色路由、状态机调度、策略决策。
- 禁止：直接执行 Git/GitHub/文件动作。

2) 执行平面（Runtime Plane）
- 负责：动作预检、执行、幂等、恢复、输出落盘。
- 禁止：业务流程编排与 UI 渲染。

3) 交互平面（Interaction Plane）
- 负责：消息总线、Section 渲染、焦点/滚动/键位交互。
- 禁止：业务规则与底层命令执行。

## 3. 术语

- Round：一次“分析 -> 建议 -> 执行 -> 回流”的闭环批次。
- Slice：巡航模式下一个时间切片窗口。
- Attempt：单个 Round 内的重规划尝试次数。
- Suggestion：可执行建议单元，必须映射到单一 Action。
- ActionResult：动作执行结果，必须包含成功性、错误语义、可恢复建议。
- RecoveryAction：失败后的标准化恢复动作（retry/skip/manual/abort）。
- TraceID：贯穿 LLM、Flow、Runtime、TUI 的链路标识。

## 4. 强约束

- 单一事实源：角色映射、协议版本、状态机定义、错误分类只能有一个权威定义点。
- 先验证后执行：配置、协议、命令、路径必须先过校验层。
- 失败可恢复：每个失败都要给出可执行恢复建议。
- 跨平台优先：Windows/Linux/macOS 行为一致性是发布门槛。
- 信息不隐藏：禁止省略号截断关键信息，默认完整换行展示。

## 5. 非目标

- 不在 V3 引入长期双运行（dual-run）架构。
- 不在 V3 内构建通用 DevOps 平台能力（超出 Git/GitHub 工作台范围）。
- 不在 V3 放宽安全边界（路径逃逸、危险命令、密钥泄露必须更严格）。

## 6. 变更治理

- 所有跨平面变更必须附带 RFC。
- 所有破坏性变更必须附带迁移与回滚方案。
- 所有状态机与协议变更必须附带 golden 测试样例。
